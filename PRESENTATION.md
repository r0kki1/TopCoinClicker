# TopCoinClicker

## Слайд 1. Название проекта
**TopCoinClicker**

Краткое описание:
Мобильная/десктопная игра-кликер на Unity с авторизацией, сохранением прогресса на сервере, таблицей лидеров и системой перевода монет между игроками.

Что говорить:
«Мой проект называется TopCoinClicker. Это клиент-серверная игра в жанре clicker. Пользователь входит в аккаунт, зарабатывает монеты кликами, улучшает силу тапа, переводит монеты другим игрокам и может видеть таблицу лидеров. Вся ключевая логика и прогресс хранятся на сервере».

---

**Цель**

- Сделать простую, понятную и расширяемую clicker-игру

- Реализовать безопасную авторизацию с JWT
- Сохранить игровой прогресс между сессиями
- Подготовить основу для дальнейшего масштабирования

Что говорить:
«Основная цель проекта была не просто сделать интерфейс с кнопкой клика, а построить полноценное клиент-серверное приложение. Я хотел, чтобы игра работала с реальной авторизацией, с сохранением данных и с возможностью расширения новыми игровыми механиками».

## Слайд 4. Архитектура
**Архитектура проекта**

- Клиент: `Unity`
- UI: `TMP_Text`, `TMP_InputField`, `Button`
- API-клиент: `HttpClient`
- Сервер: `ASP.NET Core Web API`
- База данных: `SQLite`
- Аутентификация: `JWT access token + refresh token`

Схема:

1. Unity отправляет запрос на сервер
2. Сервер проверяет токен
3. Сервер выполняет игровую логику
4. Сервер возвращает обновленные данные профиля
5. Клиент обновляет интерфейс

Что говорить:
«Архитектура разделена на клиент и сервер. Unity отвечает за UI и взаимодействие с игроком, а сервер отвечает за бизнес-логику, хранение данных и безопасность. Такой подход удобен, потому что игровую логику сложнее подделать на стороне клиента».

- Выполняет игровую логику

Что говорить:

**Игровые механики**

- Награда за тап зависит от уровня улучшения
- Стоимость следующего улучшения растет по мере прогресса
- Перевод монет проверяет баланс и валидность получателя

Также есть социальный элемент: игрок может переводить монеты другим пользователям и соревноваться в таблице лидеров».

## Слайд 8. Авторизация и безопасность
**Как работает авторизация**

- Пользователь логинится через `Top Academy`
- Сервер создает `access token`
- Сервер создает `refresh token`
- Клиент хранит токены локально
- При истечении `access token` клиент автоматически получает новый
- Если refresh token истек, пользователь перенаправляется на повторный вход

Что говорить:
«Отдельно я уделил внимание безопасности. В проекте используется схема access token и refresh token. Access token живет недолго и используется для запросов. Если он истекает, клиент автоматически обновляет его через refresh token. Это делает пользовательский опыт удобнее и при этом сохраняет защищённость сессии».

## Возможные вопросы и готовые ответы

**Почему Unity + ASP.NET, а не только Unity?**

Потому что сервер позволяет хранить прогресс, защищать данные и строить многопользовательские сценарии.

**Почему используется JWT?**

JWT удобен для stateless-аутентификации API и хорошо подходит для разделения access и refresh token.

**Зачем нужен refresh token?**

Чтобы пользователь не вылетал из аккаунта каждый раз после истечения короткоживущего access token.

**Почему логика игры вынесена на сервер?**

Чтобы нельзя было легко подменить баланс и игровое состояние на стороне клиента.

**Что в проекте самое важное технически?**

Связка Unity с backend, хранение прогресса на сервере и корректная реализация refresh token flow.
